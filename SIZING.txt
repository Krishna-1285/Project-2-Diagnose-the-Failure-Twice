------------------------------------------------------------
Introduction
------------------------------------------------------------

The purpose of this sizing analysis is to explain why the Linux Out-of-Memory (OOM) killer terminated the Ollama model server. Rather than simply stating that the application crashed, this analysis converts the kernel log evidence into measurable memory requirements using the provided memory sizing worksheet (vram_sizer.py). The objective is to determine whether the deployed model could realistically fit into the available hardware resources and to recommend a configuration that would prevent the failure.

The supplied kernel log reports that the Ollama process consumed approximately 15.99 GB of anonymous resident memory (anon-rss) before the Linux kernel invoked the OOM killer. This strongly suggests that an FP16 DeepSeek-7B model was loaded together with a large key-value (KV) cache. Since the available system memory was insufficient, the operating system protected itself by terminating the largest memory-consuming process. This analysis explains that failure using memory calculations rather than assumptions.

------------------------------------------------------------
KV Cache Analysis
------------------------------------------------------------

The supplied incident log indicates that the model server pre-allocated a KV cache using:

Context Length = 32,768 Tokens

Batch Size = 4

A larger context length dramatically increases memory consumption because the KV cache grows with every additional token retained during inference. Even when the model weights themselves fit into memory, the KV cache may exceed several additional gigabytes during long-context workloads. This explains why deployments that appear to fit on paper may still trigger Out-of-Memory events under production workloads.

Framework Overhead
------------------------------------------------------------

In addition to model weights and KV cache, inference engines require memory for activations, runtime buffers, tensor management, and framework operations. Chapter 2 recommends budgeting approximately 15–20% additional overhead beyond the calculated model memory.

This estimated total exceeds the physical memory available on a typical 16 GB workstation and explains why the Linux kernel terminated the process.

------------------------------------------------------------
Conclusion
------------------------------------------------------------

The sizing analysis confirms that the Out-of-Memory event resulted from insufficient available memory rather than an application defect. The memory consumed by the FP16 DeepSeek-7B model, combined with the large pre-allocated KV cache and framework overhead, exceeded the capacity of the host system. The Linux kernel therefore invoked the OOM killer and terminated the Ollama process to preserve overall system stability.

Deploying a Q4_K_M quantized model, reducing the context length or batch size, and configuring appropriate memory limits would prevent this failure from recurring. This analysis demonstrates that performing workload sizing before deployment is an essential systems administration practice because it allows administrators to identify resource limitations before they become production outages.
